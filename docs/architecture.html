Publishing reviewed code across an air gap, one firewall posture at a time.

edge-deploy-core is an operator-only Python package that publishes reviewed Autobench and Dispatch commits from GitHub to corporate Bitbucket, and deploys them to Hadoop Edge Nodes — through exclusive firewall postures, interactive RSA authentication, and a lossy screen-scraped terminal channel. Every release is a durable, resumable run-ledger state machine.

01Code lifecycle: who touches what

Development is normal GitHub flow — contributors never need Bitbucket, Edge access, SSH, Kerberos, or RSA credentials. Only the Release Operator crosses the boundary.

Contributor

Develops locally or through Cursor. Opens a GitHub pull request. Needs nothing beyond GitHub.

Maintainer

Reviews and squash-merges the PR after CI and human approval. GitHub main is the single source of truth.

Release Operator

Explicitly releases merged main: Publish fast-forwards the exact reviewed commit to Bitbucket, then Deploy applies it to Edge Nodes.

Edge Nodes

Run each Tool's own update.sh / install.sh. Verified for final commit, drift, permissions, and smoke checks.

Why the split? The workstation's firewall posture is exclusive: at any moment at most one of GitHub or Bitbucket-plus-Edge is writable — never both. Switching posture is a human act outside the engine's control, takes up to a minute to propagate, and everything crosses a corporate proxy that accepts TCP connects in every posture and only fails at the HTTP layer. The whole architecture flows from these constraints.

02Interactive release simulator

A Release is a run-ledger state machine advanced by short, idempotent, posture-scoped phase commands (ADR-0008). Since ADR-0012 the tag order is tag-bitbucket before tag-github, so a release needs only two posture switches: github → bitbucket+edge → github. Switch the firewall on the left, then run phases — or try guided mode, which absorbs the posture transients for you.

TCP probes lie here — the proxy accepts TCP connects in every posture. Git endpoints are probed at protocol level: git ls-remote for reads, git push --dry-run against refs/edge-deploy/posture-probe for writes (ADR-0012). Only Edge SSH stays TCP-probed.

Durability rules: at most one open Run per Tool — starting a new release while one is open is refused with the legal options (continue, abandon, scoped deploy). run.lock (PID + hostname) blocks concurrent controllers. Every phase entry compares the ledger's Engine Identity (engine.content_sha256) to the running process and refuses skew. A failed node retries with deploy --nodes node04 without re-running verify or publish.
Guided mode (ADR-0010 + ADR-0012): after you confirm a posture switch, the engine polls the probes for up to ~90 s while the switch propagates, then continues. If a phase still hits a transient remote error right after a switch (an HTTP 503 from the proxy, git ls-remote exit 128), guided mode retries the phase with 10/20/30 s backoff — safe because phases are idempotent and ledger-guarded. Ctrl+C pauses cleanly: the run stays open and the resume command is printed. Unguided invocations still fail fast.

03Inside the deploy phase — per node

Deploy fans out per node and continues-and-reports on failure (ADR-0003) — there is no auto-rollback; rollback is an explicit operator act that replays a recorded release tag through the same phase chain. Click a step to expand it, or

  1. auth.pyAuthBroker — interactive secret boundary
    Deploy is the only phase that authenticates. The operator types the RSA passcode into the controller tmux pane when prompted; the progress heartbeat shows >>> WAITING FOR OPERATOR — …. Credentials are never persisted (ADR-0002).
  2. dependencies.pyDependency bundle delivery
    Dependencies are first-class (ADR-0006, contract v1.1.0): content-addressed bundles built locally, delivered to the node, never committed. A rollout is refused outright if dependencies changed relative to what the snapshot expects (ADR-0005) — no silent partial upgrades.
  3. runner.pyCheckout update to the exact Snapshot
    The node's checkout is moved to the exact published commit. Every judgement afterwards compares against the Snapshot's tree — never the operator's working tree, which may have moved on.
  4. tool's install.shInstall trigger
    The engine invokes the Tool's own install.sh per the standard on-node deploy interface (ADR-0004). Nodes never import edge-deploy-core; the contract is files and exit codes.
  5. tool profileSmoke checks
    Smoke commands come from the committed edge_deploy.yaml Tool Profile — node-independent release metadata, no operator paths or credentials.
  6. drift.pyDrift detection
    Drift = any difference between runtime-critical files in the released commit and files on the node. Paths are enumerated from the snapshot tree, not the working tree. Evidence is written to files on the node and read back verbatim (ADR-0009) — no parsing of scrolled terminal output.

04Pane-safe transport: the hostile channel

Kerberos/RSA policy rules out scp, sftp, and SSH multiplexing. The only channel to a node is the Authenticated Pane — a psmux (Windows tmux) pane holding an RSA-authenticated SSH session. It is a lossy, screen-scraped text channel, not a byte pipe. Each corruption below caused a real production failure masquerading as a later-stage error (ADR-0011). Toggle the failure modes:

The protocol (tmux_driver.py + runner.py)

  • Single-line commands only — payloads travel as base64, decoded on the node, never raw source.
  • Files move as chunked base64 (bounded line length) appended to a staging file, decoded by a Python one-liner that is itself base64-piped — so no quoting survives or matters.
  • Interpreter resolved, not assumed — tries python3.11 → python3.10 → python3 → known sys_apps path.
  • Reads between unique sentinels with digest verification — __EDGE_RESULT_START__ / SHA-256 / __EDGE_RESULT_END__; wrap-whitespace stripped before decode; the digest matched bottom-up as an exact 64-hex line.
  • Everything staged locally is written LF-only, and runner invocations check exit codes immediately — a broken script fails at the invocation with the screen attached, not two calls later.
